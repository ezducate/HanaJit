About the name

Hana Jit — هانا جيت — is two words, and a bit of a bilingual pun. Hana is from Moroccan Arabic (Darija) ها أنا (ha ana) — “here I am.” Jit is JIT, as in a Just-In-Time compiler — but in Moroccan Arabic, جيت (jit), it also means “I arrived.” Read either way, it resolves to “here I am, a JIT compiler” or “here I am, I’ve arrived.” The installable package is named hanajit (PyPI names cannot contain spaces).

Write the Python you’d write anyway. Get native code.

from hanajit import jit, prange

@jit(cache=True, workers=8)
def energy(x):
    s = 0.0
    for i in prange(len(x)):      # auto-parallel across cores
        s += x[i] * x[i]
    return np.sqrt(s / len(x))     # scalar np.* compiles too

energy(a)                            # numpy arrays pass directly
report = energy.evolve(a)            # GA squeezes the compiled code

What's inside

CPython's own parser, LLVM's codegen. Lazy per-type specialization; unsupported code falls back to the interpreter with one warning — nothing ever breaks.

Dispatch faster than Python itself. A JITed vectorcall dispatcher makes calling a compiled function cheaper than calling an empty Python function.

NumPy where it counts. 1-D/2-D arrays, slicing (strided too), negative indexing, reshape/ravel views, compiled reductions, scipy LowLevelCallable — and a coverage inventory of what falls back.

Evolutionary post-optimization. A genetic algorithm searches semantics-preserving compilation strategies per kernel: up to 2.7x beyond -O3 on reductions and stencils, validated for equivalence, persisted to the disk cache.

Every target. CPU (Linux/Windows/Apple Silicon), CUDA PTX, AMD GCN, Intel SPIR-V, Apple Metal source, FPGA HLS export — with target="auto" picking per kernel on the current machine.

How it works — nothing hidden

One IR, many machines: CPython’s own parser, fixpoint type inference, one LLVM IR module — then retargeted to your exact CPU, four GPU code formats, or FPGA HLS export. Anything unprovable falls back to the interpreter with one warning. ~3,000 readable lines; every detail below lives in the source.

1 · The frontend: it reads the Python you actually wrote

No custom parser, no bytecode tricks: ast.parse on the real source, lazy per-signature specialization, fixpoint inference over a small lattice. Outside the lattice → transparent interpreter fallback.

2 · The fusion engine: NumPy without the temporaries

Array expressions compile to element generators that compose — ufuncs, comparisons, where/clip, even virtual arange/linspace — and fuse into one loop at the consuming reduction or store. Zero temporaries, structurally: no numba flag closes the gap.

3 · Calling in: faster than a plain Python call

The dispatcher is itself LLVM-compiled: on 3.12+ every jitted function is a HanaFunction whose vectorcall slot runs generated machine code — type check, unbox, kernel, box. New specializations install via one atomic pointer swap.

4 · One IR, many machines

CPU: a TargetMachine built from your exact host features, -O3, MCJIT. The same IR is then re-triple’d for NVPTX (PTX), AMDGPU (GCN) and SPIR-V; Metal — which has no LLVM target — gets an AST→MSL transpiler instead. FPGA has no instruction stream at all, so it’s an export: annotated IR + a Vitis HLS script, for the HLS/CIRCT flows that synthesize loops into pipelined circuits. target="auto" probes driver libraries per-kernel.

GPU scope: Hana Jit emits and assembles vendor-valid GPU code (verified by real ptxas, llvm-mc, and xcrun metal), but does not yet launch kernels on a GPU. The host-side dispatch bridge is a roadmap item. The GPU backends are a validated compiler target, not a runtime. FPGA is export-only.

5 · Caching: compile once, then never again

cache=True stores the finished object file, keyed on source + signature + host CPU + flags + versions. Warm start loads machine code directly: 79 ms vs numba’s 675 ms. Atomic writes — and writable by the optimizer below.

6 · Threads: release the GIL, then use every core

nogil=True releases the GIL around the kernel only; prange fans chunks across workers; pmap parallelizes call batches. Measured: 1.78x on 2 workers, 3.6x on 10 M4 cores — memory bandwidth is typically the limiting factor. And reduce_reassoc=True reaches numpy-class reduction speed (1.58x on a 20M sum, matching numpy’s GB/s) by vectorizing accumulators with parallel SIMD lanes — reassociation only, no global fastmath, integers bit-exact. Pass float32 arrays and it compiles native 32-bit paths — half the bytes, double the SIMD lanes: 2.7x the float64 baseline on a memory-bound reduction, at exact (bounded) f32 precision.

7 · Helpers that vanish, loops that parallelize themselves

Two ergonomics borrowed from Taichi’s kernel model — with no DSL. A small @jit helper called from another @jit function is inlined at the AST level before codegen, so cross-function calls become in-line arithmetic the optimizer and fusion engine see straight through. And @jit(parallel=True) promotes the outermost for i in range(n) to a multithreaded prange — an ordinary Python loop, now running on every core. Both are pure tree rewrites on the AST ast.parse already produced; recursion and non-straight-line helpers safely stay as calls.

8 · The experimental corner (opt-in, clearly labeled)

Two opt-in, CPU-only experiments, both documented with their limits. Structural rewrite (@jit(rewrite=True)) is a library of pattern-matched algebraic transforms — e.g. a loop summing an affine series collapses to its closed form a·n(n-1)/2 + b·n, bit-exact. It is a finite pattern library, not general program understanding (which is undecidable), and -O3 already closes the simplest of these — its value is a guaranteed O(1) form plus a home for rewrites LLVM misses.

Hyper-aggressive mode (evolve_hyper(args, confirmed=True)) evolves unsafe fp transforms — reassociation, FMA, reciprocals, approximate functions — keeping whatever is fastest that matches on a large random probe suite. The result may be wrong on untested inputs. It requires explicit confirmation, is never cached, is CPU-only, never installs a regression, and must never touch safety-critical code. A superoptimizer that doesn’t guarantee its output is a power tool with a real edge — labelled as such.

9 · The genetic optimizer: it tunes to your machine

f.evolve(args) evolves compilation strategy, not code: nine genes over pipeline knobs, per-loop unroll/vector-width metadata, and opt-in fastmath. Every gene is a semantics-preserving LLVM transform — equivalence by construction, differentially gated anyway. Fitness = wall time on your machine.

On genetic search vs. code mutation. Hana Jit is genetic: evolve() runs an evolutionary search over compilation strategies. What the safe path avoids is code-mutating genetic programming (as in STOKE), where each mutated candidate may compute a different result and equivalence must be re-proven per candidate with an SMT verifier. The safe genes are transforms LLVM already guarantees equivalent, so the search space is valid by construction. The opt-in evolve_hyper() mode does allow transforms that are not equivalence-guaranteed, validating each candidate empirically against random inputs instead — accepting that trade-off deliberately. Structure-level evolution via e-graph equality saturation is a roadmap item.
